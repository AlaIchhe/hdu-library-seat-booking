# HDU 图书馆座位预约系统 — 产品需求文档

## 1. 产品概述

### 1.1 产品简介

HDU 图书馆座位预约系统是一款帮助杭州电子科技大学学生自动预约图书馆座位的工具。系统模拟用户在微信小程序上的操作流程，支持多种认证方式和预约策略，让用户可以在个人电脑或服务器上快速、准确地完成座位预约，无需在手机上手动抢座。

### 1.2 解决的问题

- **抢座难**：热门座位在开放预约的瞬间被抢光，手动操作速度无法竞争
- **时间不便**：预约窗口通常在晚上 20:00 开放，用户不便实时守候
- **重复操作**：每天需要重复相同的预约流程，耗时且容易出错
- **信息不透明**：手机端难以快速浏览所有房间、楼层的座位分布

### 1.3 核心价值

| 价值 | 说明 |
|---|---|
| **自动化** | 一次配置，系统在预约窗口开放时自动提交请求 |
| **准时性** | 支持定时执行，精确在预约开放瞬间提交 |
| **灵活性** | 支持多种认证方式、多种座位选择策略、多种交互界面 |
| **可托管** | 可部署在云服务器上 7×24 小时无人值守运行 |

## 2. 目标用户

| 用户类型 | 特征 | 使用场景 |
|---|---|---|
| **普通学生** | 不熟悉命令行，需要简单直观的操作方式 | 每天或偶尔预约一个固定座位，希望一键完成 |
| **重度用户** | 每天必去图书馆，对座位位置有明确偏好 | 一周不同天需要不同座位，希望完全自动化 |
| **运维爱好者** | 有自己的服务器或 NAS，习惯自动化运维 | 将预约脚本部署在服务器上长期运行 |
| **终端用户** | 偏好命令行操作，追求效率和控制力 | 在终端中管理多个预约方案，灵活切换 |

## 3. 功能需求

### 3.1 用户认证

系统应支持以下认证方式，用户可根据自身情况选择：

#### FR-1.1 Cookie 免密认证

- 用户从手机微信小程序中抓取一次 Cookie 字符串，粘贴到系统中即可使用
- 无需输入学号和密码，隐私性最佳
- Cookie 有过期时间，过期后需重新抓取

#### FR-1.2 密码登录认证

- 用户输入学号和密码，系统自动登录慧图平台
- 登录成功后系统记住登录状态，下次启动时免登录
- 支持交互式密码输入（密码不回显在屏幕上）

#### FR-1.3 浏览器模拟登录

- 系统启动真实浏览器自动完成登录操作
- 模拟用户在微信小程序中的操作行为，最大程度降低被检测风险
- 登录完成后自动提取凭证用于后续操作

---

### 3.2 房间与座位浏览

#### FR-2.1 房间类型查询

- 系统应展示所有可预约的房间类型（自习室、教师休息室、阅览室、讨论室等）
- 每种房间类型应能进一步查看详细信息

#### FR-2.2 楼层与座位查看

- 用户选择房间类型后，系统应展示该房间的所有楼层
- 用户选择楼层后，系统应展示该楼层的所有可用座位编号
- 系统应展示每个房间的开放时间范围（最早可预约时间和最晚可预约时间）

#### FR-2.3 座位信息缓存

- 系统应在后台预加载房间和座位信息，减少用户等待时间
- 信息加载期间应有进度提示

---

### 3.3 预约方案管理

「预约方案」是系统的基本操作单位，一个方案包含以下信息：

| 要素 | 说明 | 示例 |
|---|---|---|
| 房间类型 | 自习室 / 阅览室等 | 自习室 |
| 楼层 | 具体楼层名称 | 四楼宋韵云图 |
| 座位号 | 一个或多个座位号 | 130 座 |
| 开始时间 | 预约开始的具体时间 | 2025-01-10 13:00 |
| 使用时长 | 预约持续的小时数 | 9 小时 |
| 预约人 | 使用该座位的用户 | 张三 |

#### FR-3.1 创建预约方案

- 系统应提供交互式引导，帮助用户逐步选择房间 → 楼层 → 时间 → 座位
- 系统应校验用户输入的合法性：
  - 开始时间是否在房间开放时间范围内
  - 使用时长是否不超过房间剩余开放时间
  - 座位号是否在所选楼层中存在
- 创建过程中用户可随时取消操作

#### FR-3.2 查看方案列表

- 系统应以清晰的表格形式展示所有已创建的预约方案
- 表格应包含：序号、房间名、楼层名、座位号、开始时间、使用时长、预约人

#### FR-3.3 修改方案

- 用户应能批量修改方案中的预约开始时间和使用时长
- 支持选中部分方案修改，也支持一键全选修改

#### FR-3.4 删除方案

- 用户应能删除不需要的方案
- 支持一次删除多个方案（多选）

#### FR-3.5 方案持久化

- 所有方案应自动保存，重启系统后无需重新创建
- 方案修改后应立即持久化

---

### 3.4 预约提交

#### FR-4.1 立即预约

- 用户可随时执行「立即预约」，系统按方案列表顺序依次提交预约请求
- 用户可设定最大尝试次数和重试间隔
- 任何一个方案预约成功后，系统展示预约结果并结束
- 预约失败时展示失败原因，继续尝试下一个方案

#### FR-4.2 定时预约

- 用户可指定一个未来的精确时间点，系统在该时刻自动开始提交预约
- 等待期间实时显示当前时间、目标时间和剩余等待时间
- 等待期间用户可随时取消

#### FR-4.3 智能重试

- 系统应根据服务器返回的不同错误消息，自动做出不同决策：

| 服务器反馈 | 含义 | 系统行为 |
|---|---|---|
| 超出可预约时间范围 | 预约窗口尚未开放 | 等待后自动重试 |
| 已有预约，请勿重复 | 用户已有有效预约 | 放弃当前计划 |
| 座位无法预约 | 座位已被他人锁定 | 放弃当前方案，尝试下一个 |
| 非法请求 | 系统规则可能已更新 | 立即停止，提示用户检查更新 |

#### FR-4.4 预览模式

- 用户可开启「预览模式」，系统生成完整的预约请求但不实际提交
- 用于验证预约参数是否正确，避免因参数错误导致封号

---

### 3.5 多场景预约策略

系统应支持多种预约策略，用户可根据需求选择：

#### FR-5.1 固定座位模式

- 用户明确指定房间、楼层、座位号和预约时间
- 系统直接按此配置提交预约
- 适用于每天去同一座位的用户

#### FR-5.2 按星期自动切换

- 用户可为周一至周日分别配置不同的座位偏好
- 系统自动识别「后天是星期几」，并应用对应的配置
- 每天可独立设置：座位区域、开始时间、使用时长、是否启用
- 若当天配置为「不启用」，系统自动跳过
- 适用于一周不同天在不同区域上课/自习的用户

#### FR-5.3 范围随机选择

- 用户指定一个座位号范围（如 100-150 号），系统在范围内随机选取
- 前几次尝试倾向于用户偏好的座位，后续尝试随机分散
- 适用于对座位位置不敏感、只想抢到该区域任意座位的用户

#### FR-5.4 多候选时间策略

- 当用户指定的时间段无可用座位时，系统自动尝试替代时间段：
  - 首先尝试用户指定的完整时段
  - 若失败，尝试目标时段的第一个小时
  - 若失败，尝试目标日早上 8:00
  - 若失败，尝试当前最近的可用时段
- 适用于预约窗口拥挤、需要灵活应对的用户

---

### 3.6 通知与反馈

#### FR-6.1 实时日志

- 系统应在执行过程中实时展示日志：查询进度、提交状态、错误信息
- 日志应包含时间戳，便于追踪

#### FR-6.2 预约结果展示

- 预约成功时，应以清晰格式展示完整信息：
  - 房间名、楼层名、座位号
  - 开始时间、使用时长
  - 预约人姓名
- 预约失败时，应明确说明失败原因

#### FR-6.3 微信消息推送

- 系统可选择性接入微信消息推送服务
- 预约成功或失败后，自动向用户微信发送通知
- 用户无需守在电脑前也能获知预约结果

---

### 3.7 用户界面

系统应提供多种交互界面，用户可根据偏好选择：

#### FR-7.1 网页控制台

- 通过浏览器访问的图形化操作界面
- 表单式参数填写：房间类型、楼层 ID、座位号、开始小时、使用时长、执行时间等
- 一键「载入配置」读取已有设置
- 一键「保存计划」将当前设置写回
- 一键「开始执行」启动预约
- 一键「取消任务」中止运行中的预约
- 实时展示执行日志

#### FR-7.2 终端交互界面

- 在命令行中提供菜单驱动的交互体验
- 数字菜单导航（如「1. 管理方案」「2. 修改时间」「3. 立即抢座」）
- 表格形式展示数据（方案列表、预约结果等）
- 重要信息使用高亮颜色显示（如错误消息红色）
- 等待操作显示进度动画

#### FR-7.3 命令行一次性执行

- 支持通过命令行参数直接指定所有预约参数
- 运行后自动完成预约流程并退出
- 适用于脚本调用或 cron 定时任务
- 也可通过环境变量配置全部参数，无需任何交互

## 4. 用户工作流

### 场景 A：首次使用，预约固定座位

```
1. 用户选择 Cookie 认证，从手机抓取 Cookie
2. 用户填写预约参数：房间、楼层、座位号、时间、时长
3. 用户点击「开始执行」
4. 系统查询房间信息 → 查询座位布局 → 定位目标座位
5. 若到达预约开放时间 → 立即提交
6. 若未到开放时间 → 自动等待到点后提交
7. 展示预约结果
```

### 场景 B：按星期自动预约（服务器部署）

```
1. 用户配置一周七天的座位偏好（周一三楼、周二四楼...）
2. 用户配置服务器 cron 定时任务（每天固定时间运行）
3. 用户设置微信通知（可选）
4. 系统每天自动运行：
   a. 自动登录
   b. 判断后天是星期几
   c. 若该天已启用 → 按配置抢座
   d. 若该天未启用 → 自动退出
5. 通过微信推送结果给用户
```

### 场景 C：管理多个预约方案

```
1. 用户使用密码登录
2. 系统后台加载所有房间和座位信息
3. 用户在菜单中创建多个方案（如方案A：三楼早8点，方案B：四楼下午1点）
4. 用户选择立即执行，系统依次尝试提交
5. 哪个先成功就用哪个
```

### 场景 D：定时抢座

```
1. 用户已配置好预约方案
2. 用户选择「定时执行」，输入目标时间（如 19:59:30）
3. 系统显示倒计时，等待中
4. 到达目标时间后自动开始提交预约
5. 展示结果
```

## 5. 安全与风险提示

### 5.1 账号安全

- 密码或 Cookie 存储在用户本地设备上
- 用户应妥善保管配置文件，避免泄露给他人
- 使用 Cookie 方式无需提供密码，安全性更优

### 5.2 封号风险

- 过于频繁的预约请求可能导致图书馆账号被封禁
- 系统默认设置了安全的请求间隔
- 用户在修改请求频率时应谨慎，建议不低于 5 秒

### 5.3 使用合规

- 本系统为非官方工具，通过模拟用户操作实现自动化
- 图书馆平台更新可能导致系统暂时不可用
- 用户应遵守图书馆使用规定，合理使用自动化工具

---

## 6. 运行环境

### 6.1 个人电脑

- 支持 Windows、macOS、Linux
- 部分功能（如终端窗口最大化）仅 Windows 支持
- 使用网页控制台或终端界面

### 6.2 云服务器

- 支持主流 Linux 发行版
- 可配合 cron 定时任务实现全自动无人值守
- 浏览器模拟登录模式需要服务器安装浏览器
